# ccm v1.2 — sessions across accounts, compose layer, insight (design spec)

Date: 2026-07-06 · Status: approved feature list from user, implemented same day

Eight features on top of the v1.0 profile-launcher architecture.

## move-session (+ in-session recommendation)

`ccm move-session <to> [session-id] [--no-launch]` finds the newest transcript for the
current folder's project slug (slug = absolute path, non-alphanumerics → `-`) across all
profiles and the default `~/.claude`, copies it plus per-session artifacts (file-history,
sessions, session-data, tasks, todos entries containing the session UUID) into the target
profile, and relaunches with `--resume <id>`. Copy, not move — the original stays on the
source account. The statusline appends `→ ccm move-session <best>` when the current
account's tightest window is ≥90% and another profile has ≥30 headroom; quota toasts
carry the same tip. This keeps switching an explicit user action (ToS posture unchanged).

## Compose layer (settings overrides, CLAUDE.md layering, MCP injection)

`composeProfile(name, dir)` runs at every launch, replacing the old copy-if-newer sync:

- **settings.json** = deepMerge(shared settings, `~/.ccm/overrides/<name>.json`).
  Objects merge recursively; arrays/scalars replace. Managed via
  `ccm override <name> key=value | --unset key | --clear` (dotted keys, JSON-parsed values).
- **CLAUDE.md** = shared CLAUDE.md + `<!-- ccm: profile … -->` + `overrides/<name>.CLAUDE.md`
  when the fragment is non-empty.
- Both use a newer-mtime gate (max of source mtimes vs profile file) so `/config` edits
  made inside a session survive until a source actually changes.
- **MCP**: `~/.ccm/shared/mcp.json` (seeded once from `~/.claude/.mcp.json`) is injected
  into each profile's `.mcp.json`. A per-profile ownership record
  (`cache/mcp-injected/<name>.json`) tracks which server names ccm owns: unshared servers
  are removed on next launch, and a same-named server the profile added itself always
  wins (warn, never clobber). `ccm mcp list | share <name> [--from <profile>] | unshare`.

## Quota-aware picker

Rows sorted most-headroom-first (headroom = 100 − max window percent; unknown data last,
stable order), ✦ marker on the best, quick refresh of stale cache capped at 1.2 s before
rendering so the picker never blocks on the network.

## Quota notifications

Bucket model per window: 0 (<80), 1 (80–94), 2 (≥95). Toast fires only on an upward
bucket crossing, or on a drop to 0 from ≥1 ("fresh again") — no repeats while a bucket
holds. Checks run inside `ccm refresh`, which the statusline already spawns when its
cache is stale, so monitoring is active exactly while sessions are open (no daemon).
Toasts via PowerShell WinRT using PowerShell's AppUserModelID (no app registration).
`ccm notify on|off|status|test`; enabled by default, stored in registry config.

## Windows Terminal integration

`ccm wt install|uninstall|status` writes a WT *fragment*
(`%LOCALAPPDATA%\Microsoft\Windows Terminal\Fragments\ccm\ccm.json`) — the sanctioned
mechanism; never edits the user's WT settings.json (which allows comments). One profile
per account, `commandline: ccm <name>`, tabColor from the account's registry color.
Auto-regenerated on `ccm add/import/remove` when installed. WT detection handles Store
installs where the LocalState folder doesn't exist yet (package dir / wt.exe shim).

## ccm doctor

Checks: claude on PATH (+version), ccm home, registry parse, statusline installed, WT
fragment sync, per profile: dir exists, credentials/token expiry, identity known,
junction health (auto-recreates missing junctions), stale lock cleanup, active overrides;
plus one live usage-API probe and orphan profile-dir detection. Exit 1 only on hard
failures.

## Guards

Command words are reserved as profile names. All new pure logic (deepMerge, dotted
set/unset, MCP merge, buckets/diff, headroom/bestAlternative, slug, fragment build,
picker sort) is unit-tested; 32 tests total.
